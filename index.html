Introduction & Workshop Goals

This workshop focuses on practical actions researchers can take to make their data more manageable, findable, and secure. Rather than policy compliance, we emphasize skills you can apply immediately.

By the end of this session, you will be able to:

  1. Establish and implement a consistent file naming convention
  2. Create effective documentation including README files and data dictionaries
  3. Apply appropriate metadata standards for your discipline
  4. Implement the 3-2-1 backup strategy for data protection
  5. Secure sensitive data using encryption and access controls
  6. Use version control to track changes to your data and code
  7. Prepare data for sharing by selecting formats and de-identification methods
Workshop Format: 2–3 hours total (sections can be delivered independently)
Approach: Short concept introductions + hands-on activities
Materials: Bring laptops and sample data files from current projects

Section 1: File Organization & Naming Conventions

Why This Matters

Poor file organization leads to:

  • Wasted time searching for files
  • Accidental data loss or overwriting
  • Version confusion ("final_v2_FINAL_edited.docx")
  • Difficulty reproducing your own work
  • Collaboration breakdowns
A consistent naming system allows you to identify files at a glance, locate data quickly, and collaborate effectively with others—including your future self.

Elements to Include in File Names

Core Elements

  • Project identifier: Short code or abbreviation
  • Date: YYYYMMDD or YYYY-MM-DD
  • Description: Brief, meaningful descriptor

Optional Elements

  • Version number: v01, v02 (leading zeros)
  • Creator initials: For collaborative projects
  • Sequential numbers: 001, 002, 003
Kay P Maye | Tulane University Libraries

File Naming Best Practices

The Rules

  • Keep names under 40–50 characters when possible
  • Avoid spaces – use underscores (_) or hyphens (-) instead
  • Avoid special characters: ~ ! @ # $ % ^ & * ( ) ` ; : < > ? . , [ ] { } ' " |
  • Use only alphanumeric characters (A-Z, a-z, 0-9)
  • Be consistent with capitalization (choose one style)
  • Document your convention in a README file

Example Conventions

# Survey data: PROJ_Survey_ParticipantID_YYYYMMDD.csv Example: ClimateStudy_Survey_P042_20260115.csv # Lab images: YYYYMMDD_ExperimentID_SampleID_Condition_###.tif Example: 20260115_EXP03_S12_Control_001.tif # Documents: ProjectAbbr_DocType_YYYYMMDD_v##.docx Example: GenomeProj_Protocol_20260115_v02.docx

Example Folder Structure

ProjectName/ ├── README.txt ├── 01_RawData/ (original, unmodified data) ├── 02_ProcessedData/ (cleaned, transformed data) ├── 03_Analysis/ (scripts, statistical output) ├── 04_Documentation/ (protocols, codebooks) ├── 05_Outputs/ (figures, tables, reports) └── Archive/ (old versions, deprecated files)

Section 2: Creating Effective Documentation

Why Documentation Matters

Documentation enables others (including your future self) to understand, interpret, and reuse your data. Without proper documentation, data loses its meaning and value over time.

Key Principle: Start documenting as you collect data, not at the end of the project!

Types of Documentation

README Files

The first file anyone should read when encountering your dataset. Provides context and navigation.

Format: Plain text (.txt) or Markdown (.md) for maximum accessibility.

Data Dictionaries / Codebooks

Define every variable/column in your dataset. Essential for tabular data (spreadsheets, databases).

Format: Spreadsheet or plain text that accompanies your data files.

Other Documentation Types

  • Protocols: Step-by-step procedures for data collection
  • Lab notebooks: Detailed records of experiments
  • Analysis scripts: Commented code explaining each step
  • Changelogs: Record of modifications to files over time
Kay P Maye | Tulane University Libraries

README Files & Data Dictionaries

README Contents Checklist

  • ☐ Title and brief description
  • ☐ Creator(s) with contact info & ORCID
  • ☐ Date(s) of data collection
  • ☐ Methods and instruments used
  • ☐ File list with descriptions
  • ☐ File naming convention explanation
  • ☐ Software requirements & versions
  • ☐ Related publications
  • ☐ License/terms of use
  • ☐ Funding acknowledgments

Section 3: Applying Metadata Standards

What Is Metadata?

Metadata is "data about data" – structured information that describes, explains, locates, and makes it easier to retrieve, use, and manage data.

Think of it like the label on a jar: It tells you what's inside, when it was made, and how to use it.

Types of Metadata

D
Descriptive
Title, creator, subject, keywords – helps people find your data
S
Structural
File format, relationships between files – helps systems process your data
A
Administrative
Creation date, access rights, preservation information
T
Technical
File size, software requirements, compression methods
Kay P Maye | Tulane University Libraries

Finding Your Discipline's Metadata Standard

Using a community-accepted metadata standard makes your data more interoperable and discoverable.

Common Standards by Discipline

Discipline Standard Use Case
General / Cross-disciplinary Dublin Core, DataCite Basic descriptive metadata for any dataset
Social Sciences DDI (Data Documentation Initiative) Survey data, questionnaires, statistical data
Ecology EML (Ecological Metadata Language) Environmental and ecological datasets
Biology Darwin Core Biodiversity, species observations, specimens
Geospatial ISO 19115, FGDC-CSDGM GIS data, maps, spatial information
Neuroimaging BIDS MRI, fMRI, EEG data organization
Humanities TEI, VRA Core Text encoding, visual resources

Where to Find Standards

  • RDA Metadata Standards Catalog
  • FAIRsharing.org
  • DCC Disciplinary Metadata
  • Check what your target repository requires
  • Ask your subject librarian!

Section 4: Data Security and Backup

Why Security Matters

Research data can be lost through:

  • Hardware failure: Hard drives fail, laptops get stolen
  • Human error: Accidental deletion, overwriting files
  • Natural disasters: Floods, fires, power surges
  • Cyber threats: Ransomware, unauthorized access
  • Software issues: Corruption, incompatibility
The Question Isn't IF Your Data Will Be at Risk—It's WHEN
Proactive planning prevents catastrophic loss.

Two Key Strategies

🔄 Backup

Creating copies of your data in multiple locations so you can recover if something goes wrong.

Protects against: loss, corruption, hardware failure

🔒 Security

Protecting your data from unauthorized access, modification, or disclosure.

Protects against: theft, breaches, compliance violations

Kay P Maye | Tulane University Libraries

The 3-2-1 Backup Rule

This proven strategy protects against data loss from hardware failure, theft, natural disasters, and ransomware:

3

Copies of your data
(1 original + 2 backups)

2

Different media types
(e.g., local drive + cloud)

1

Copy offsite
(cloud, remote server, or physically separate location)

Implementation Example

  1. Copy 1 (Original): Your working files on your computer or institutional server
  2. Copy 2 (Local backup): External hard drive, NAS device, or different institutional server
  3. Copy 3 (Offsite backup): Cloud storage (institutional or commercial), remote server
Best Practices: Automate backups • Test restores regularly • Keep raw data read-only • Document your backup strategy
Kay P Maye | Tulane University Libraries

Encryption & Access Control

Types of Encryption

🔐 Encryption at Rest

Protects data stored on devices (hard drives, USB drives)

Tools:

  • Windows: BitLocker
  • Mac: FileVault
  • Linux: LUKS
  • Cross-platform: VeraCrypt

🔄 Encryption in Transit

Protects data being transferred (uploads, emails)

Methods:

  • HTTPS websites
  • SFTP file transfers
  • VPN connections
  • Encrypted email
⚠️ Critical Warning: If you forget your encryption password, your data may be permanently inaccessible. Store passwords securely using a password manager (LastPass, 1Password, Bitwarden, KeePass).

Access Control Principles

  • Principle of least privilege: Give collaborators only the access they need
  • Use unique accounts: Avoid shared logins to track who accessed what
  • Enable multi-factor authentication (MFA): Adds security beyond passwords
  • Review access periodically: Remove access when team members leave

Section 5: Version Control

Why Version Control?

Version control tracks changes to files over time, allowing you to see what changed, when, and by whom. It prevents overwriting, enables recovery of previous versions, and supports collaboration without conflicts.

Simple: File Naming

For straightforward projects:

  • Include version number: document_v01.docx
  • Use dates: analysis_20260115.R
  • Keep a changelog
  • Archive old versions (don't delete)
Version | Date | Author | Changes v01 | 2026-01-10 | JDS | Initial draft v02 | 2026-01-15 | JDS | Added methods v03 | 2026-01-18 | MKT | Revised stats

Advanced: Git

For code and complex projects:

  • Repository: Project folder tracked by Git
  • Commit: Snapshot at a point in time
  • Branch: Parallel version for experiments
  • Remote: Copy on GitHub/GitLab

Getting Started:

  • Install Git
  • Try GitHub Desktop (GUI)
  • Software Carpentry Tutorial
When to use Git: Code, scripts, plain text files, configuration files
When to use file naming: Word documents, images, large binary files

Section 6: Preparing Data for Sharing

File Format Selection

Choose file formats that maximize long-term accessibility and reusability:

Data Type Preferred Formats Avoid
Tabular data CSV, TSV Excel (.xlsx) for archiving
Text Plain text (.txt), Markdown (.md), PDF/A Proprietary formats
Images TIFF (uncompressed), PNG JPEG for scientific images
Audio WAV, FLAC (lossless) MP3 (lossy compression)
Video MP4 (H.264), uncompressed AVI Proprietary codecs
Code Plain text source files (.R, .py, .m) Compiled binaries only
Format Conversion Tips:
  • Keep original files in addition to converted versions
  • Document any conversion process or settings used
  • Verify converted files maintain all necessary information
Kay P Maye | Tulane University Libraries

De-identification of Human Subjects Data

If sharing data collected from human participants, you must remove or mask identifying information. Consult your IRB for specific requirements.

Direct Identifiers (Always Remove)

  • Names, addresses, phone numbers
  • Email addresses
  • Social Security numbers
  • Medical record numbers
  • Biometric identifiers (photos)
  • IP addresses, account numbers

Indirect Identifiers (May Need Action)

  • Dates → use age ranges or relative dates
  • Geography smaller than state → aggregate
  • Age over 89 → aggregate to 90+
  • Rare occupations → broader categories
  • Race/ethnicity in small samples

De-identification Techniques

Technique Description Example
Removal Delete the variable entirely Remove "Name" column
Suppression Replace values with blanks or codes "John Smith" → "P001"
Generalization Reduce precision Age 34 → Age range 30-39
Perturbation Add random noise Salary ± random amount
k-anonymity Ensure each combo appears k+ times At least 5 people per cell
